## Overview

Deploying Task Pro API to production requires careful attention to security, performance, reliability, and monitoring. This guide covers production-ready configuration and best practices.

## Pre-Deployment Checklist

<Steps>

## Production Environment Variables

### Required Configuration

```bash
# Application Profile
SPRING_PROFILES_ACTIVE=prod

# Database Connection
SPRING_DATASOURCE_URL=jdbc:postgresql://prod-db.example.com:5432/task_pro_production?ssl=true&sslmode=require
SPRING_DATASOURCE_USERNAME=task_pro_prod_user
SPRING_DATASOURCE_PASSWORD=<strong_random_password_from_secrets_manager>

# Schema Management (CRITICAL: Never use 'update' in production)
SPRING_JPA_HBM2DDL=validate

# CORS (Specific origins only)
APP_CORS_ALLOWED_ORIGINS=https://app.example.com,https://www.example.com

# JVM Memory Settings (adjust based on instance size)
JAVA_TOOL_OPTIONS="-Xms1g -Xmx2g"
```

<Warning>
**Critical Settings:**

- Never use `SPRING_JPA_HBM2DDL=update` or `create` in production
- Never use `APP_CORS_ALLOWED_ORIGINS=*` in production
- Always use SSL for database connections
- Always use `SPRING_PROFILES_ACTIVE=prod`
</Warning>

## Database Production Setup

### Connection Pooling

Configure HikariCP for optimal performance:

```yaml
# application-prod.yml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20      # Max connections (adjust based on traffic)
      minimum-idle: 5            # Keep 5 connections ready
      connection-timeout: 30000  # 30 seconds
      idle-timeout: 600000       # 10 minutes
      max-lifetime: 1800000      # 30 minutes
      leak-detection-threshold: 60000  # Detect leaks after 1 minute
```

**Sizing guidelines:**

- **Small apps (< 100 concurrent users):** 10-20 connections
- **Medium apps (100-1000 users):** 20-50 connections
- **Large apps (> 1000 users):** 50-100+ connections

<Tip>
Formula: `connections = ((core_count * 2) + effective_spindle_count)`

For modern SSDs, use: `connections = (core_count * 4)`
</Tip>

### Database Indexes

Ensure these indexes exist for optimal performance:

```sql
-- User lookups
CREATE INDEX IF NOT EXISTS idx_users_email ON users(email);
CREATE INDEX IF NOT EXISTS idx_users_username ON users(username);

-- Task queries
CREATE INDEX IF NOT EXISTS idx_tasks_user_id ON tasks(user_id);
CREATE INDEX IF NOT EXISTS idx_tasks_folder_id ON tasks(folder_id);
CREATE INDEX IF NOT EXISTS idx_tasks_status ON tasks(status);
CREATE INDEX IF NOT EXISTS idx_tasks_due_date ON tasks(due_date);
CREATE INDEX IF NOT EXISTS idx_tasks_user_status ON tasks(user_id, status);
CREATE INDEX IF NOT EXISTS idx_tasks_user_folder ON tasks(user_id, folder_id);

-- Subtask queries
CREATE INDEX IF NOT EXISTS idx_subtasks_task_id ON subtasks(task_id);
CREATE INDEX IF NOT EXISTS idx_subtasks_status ON subtasks(status);

-- Folder queries
CREATE INDEX IF NOT EXISTS idx_folders_user_id ON folders(user_id);
CREATE INDEX IF NOT EXISTS idx_folders_parent_id ON folders(parent_folder_id);

-- Tag queries
CREATE INDEX IF NOT EXISTS idx_tags_user_id ON tags(user_id);
CREATE INDEX IF NOT EXISTS idx_tags_name ON tags(user_id, tag_name);

-- Timestamp queries
CREATE INDEX IF NOT EXISTS idx_tasks_created_at ON tasks(created_at);
CREATE INDEX IF NOT EXISTS idx_tasks_updated_at ON tasks(updated_at);
```

### Automated Backups

**Daily backup script:**

```bash
#!/bin/bash
# /opt/scripts/backup-task-pro.sh

DATE=$(date +%Y%m%d_%H%M%S)
BACKUP_DIR="/backups/task-pro"
DB_NAME="task_pro_production"
DB_USER="task_pro_prod_user"
DB_HOST="prod-db.example.com"
RETENTION_DAYS=30

# Create backup directory
mkdir -p "$BACKUP_DIR"

# Perform backup
pg_dump -h "$DB_HOST" -U "$DB_USER" -d "$DB_NAME" -F c -f "$BACKUP_DIR/backup_${DATE}.dump"

# Compress backup
gzip "$BACKUP_DIR/backup_${DATE}.dump"

# Remove old backups
find "$BACKUP_DIR" -name "backup_*.dump.gz" -mtime +$RETENTION_DAYS -delete

# Upload to S3 (optional)
# aws s3 cp "$BACKUP_DIR/backup_${DATE}.dump.gz" s3://my-bucket/backups/task-pro/

echo "Backup completed: backup_${DATE}.dump.gz"
```

**Cron job:**

```bash
# Run daily at 2 AM
0 2 * * * /opt/scripts/backup-task-pro.sh >> /var/log/task-pro-backup.log 2>&1
```

## Security Best Practices

### 1. Secrets Management

<Tabs>
  <Tab title="AWS Secrets Manager">
    ```bash
    # Store secrets
    aws secretsmanager create-secret \
      --name task-pro/prod/db-password \
      --secret-string "your_strong_password"
    
    # Retrieve in application startup script
    export SPRING_DATASOURCE_PASSWORD=$(
      aws secretsmanager get-secret-value \
        --secret-id task-pro/prod/db-password \
        --query SecretString \
    ```

  </Tab>
  
  <Tab title="HashiCorp Vault">
    ```bash
    # Store secrets
    vault kv put secret/task-pro/prod \
      db-password="your_strong_password" \
      jwt-secret="your_jwt_secret"
    
    # Retrieve in application
    export SPRING_DATASOURCE_PASSWORD=$(
      vault kv get -field=db-password secret/task-pro/prod
    )
    ```

  <Tab title="Kubernetes Secrets">

### 2. HTTPS/TLS Configuration

**Use a reverse proxy (recommended):**

<CodeGroup>
```nginx Nginx
server {
    listen 443 ssl http2;
    server_name api.example.com;
    
    ssl_certificate /etc/ssl/certs/api.example.com.crt;
    ssl_certificate_key /etc/ssl/private/api.example.com.key;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    
    location / {
        proxy_pass http://localhost:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
    
    location /actuator/health {
        proxy_pass http://localhost:8080/actuator/health;
        access_log off;
    }
}

# Redirect HTTP to HTTPS
server {
    listen 80;
    server_name api.example.com;
    return 301 https://$server_name$request_uri;
}
```

```apache Apache
<VirtualHost *:443>
    ServerName api.example.com
    
    SSLEngine on
    SSLCertificateFile /etc/ssl/certs/api.example.com.crt
    SSLCertificateKeyFile /etc/ssl/private/api.example.com.key
    SSLProtocol all -SSLv2 -SSLv3 -TLSv1 -TLSv1.1
    SSLCipherSuite HIGH:!aNULL:!MD5
    
    ProxyPreserveHost On
    ProxyPass / http://localhost:8080/
    ProxyPassReverse / http://localhost:8080/
    
    RequestHeader set X-Forwarded-Proto "https"
    RequestHeader set X-Forwarded-Port "443"
</VirtualHost>

<VirtualHost *:80>
    ServerName api.example.com
    Redirect permanent / https://api.example.com/
</VirtualHost>
```

```yaml Traefik
# docker-compose.yml
services:
  traefik:
    image: traefik:v2.10
    command:
      - "--providers.docker=true"
      - "--entrypoints.web.address=:80"

      - "--certificatesresolvers.le.acme.email=admin@example.com"
      - "--certificatesresolvers.le.acme.storage=/letsencrypt/acme.json"
      - "--certificatesresolvers.le.acme.httpchallenge.entrypoint=web"
    ports:
      - "80:80"

  api:

      - "traefik.enable=true"

      - "traefik.http.routers.api.entrypoints=websecure"
      - "traefik.http.routers.api.tls.certresolver=le"
```
</CodeGroup>

### 3. Rate Limiting

**Nginx rate limiting:**

```nginx
http {
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;
    
    server {
        location /api/ {
            limit_req zone=api_limit burst=20 nodelay;
            proxy_pass http://localhost:8080;
        }
    }
}
```

### 4. Firewall Rules

```bash
# Allow only necessary ports
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow 22/tcp   # SSH
sudo ufw allow 80/tcp   # HTTP
sudo ufw allow 443/tcp  # HTTPS
sudo ufw enable

# Application port should NOT be exposed directly
# Use reverse proxy instead
```

### Alerting

**Example Prometheus alerts:**

```yaml
groups:
  - name: task-pro-api
    rules:
      - alert: HighErrorRate
        expr: rate(http_server_requests_seconds_count{status=~"5.."}[5m]) > 0.05
        for: 5m
        annotations:
          summary: "High error rate detected"
          description: "Error rate is {{ $value }} per second"
      
      - alert: HighMemoryUsage
        expr: jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} > 0.9
        for: 10m
        annotations:
          summary: "High memory usage"
          description: "Heap usage is at {{ $value }}%"
      
      - alert: ServiceDown
        expr: up{job="task-pro-api"} == 0
        for: 1m
        annotations:
          summary: "Task Pro API is down"
          description: "The API has been down for more than 1 minute"
```

## Scaling Strategies

### Horizontal Scaling

**Load balancer configuration:**

```nginx
upstream task_pro_api {
    least_conn;
    server api1.internal:8080;
    server api2.internal:8080;
    server api3.internal:8080;
}

server {
    listen 443 ssl;
    server_name api.example.com;
    
    location / {
        proxy_pass http://task_pro_api;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

**Kubernetes Horizontal Pod Autoscaler:**

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: task-pro-api-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: task-pro-api
  minReplicas: 3
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
```

### Vertical Scaling

**Adjust resources based on load:**

```bash
# Small workload (< 100 users)
JAVA_TOOL_OPTIONS="-Xms512m -Xmx1g"
CPU: 1 core
Memory: 2GB

# Medium workload (100-1000 users)
JAVA_TOOL_OPTIONS="-Xms1g -Xmx2g"
CPU: 2 cores
Memory: 4GB

# Large workload (> 1000 users)
JAVA_TOOL_OPTIONS="-Xms2g -Xmx4g"
CPU: 4+ cores
Memory: 8GB+
```

## Deployment Workflow

<Steps>
  <Step title="Build and test">
    ```bash
    # Run tests
    ./gradlew test
    
    # Build JAR
    ./gradlew bootJar
    
    # Build Docker image
    docker build -t task-pro-api:v1.2.3 .
    ```
  </Step>
  
  <Step title="Tag and push">
    ```bash
    # Tag image
    docker tag task-pro-api:v1.2.3 registry.example.com/task-pro-api:v1.2.3
    docker tag task-pro-api:v1.2.3 registry.example.com/task-pro-api:latest
    
    # Push to registry
    docker push registry.example.com/task-pro-api:v1.2.3
    docker push registry.example.com/task-pro-api:latest
    ```
  </Step>
  
  <Step title="Deploy to staging">
    ```bash
    # Update staging environment
    kubectl set image deployment/task-pro-api-staging \
      api=registry.example.com/task-pro-api:v1.2.3 \
      --namespace=staging
    
    # Verify deployment
    kubectl rollout status deployment/task-pro-api-staging -n staging
    ```
  </Step>
  
  <Step title="Run smoke tests">
    ```bash
    # Test health endpoint
    curl https://staging-api.example.com/actuator/health
    
    # Test authentication
    curl -X POST https://staging-api.example.com/api/auth/login \
      -H "Content-Type: application/json" \
      -d '{"email":"test@example.com","password":"test123"}'
    ```
  </Step>
  
  <Step title="Deploy to production">
    ```bash
    # Update production with rolling update
    kubectl set image deployment/task-pro-api \
      api=registry.example.com/task-pro-api:v1.2.3 \
      --namespace=production
    
    # Monitor rollout
    kubectl rollout status deployment/task-pro-api -n production
    ```
  </Step>
  
  <Step title="Monitor and verify">
    - Check health endpoints
    - Monitor error rates
    - Verify database connections
    - Test critical user flows
  </Step>
  
  <Step title="Rollback if needed">
    ```bash
    # Rollback to previous version
    kubectl rollout undo deployment/task-pro-api -n production
    ```
  </Step>
</Steps>

## Common Production Issues

<AccordionGroup>
  <Accordion title="High memory usage / OutOfMemoryError">
    **Symptoms:** Application crashes, slow response times
    
    **Solutions:**
    1. Increase JVM heap: `JAVA_TOOL_OPTIONS="-Xms2g -Xmx4g"`
    2. Check for memory leaks with profiler
    3. Reduce connection pool size if too large
    4. Scale horizontally instead of vertically
  </Accordion>
  
  <Accordion title="Slow database queries">
    **Symptoms:** Increased response times, timeouts
    
    **Solutions:**
    1. Add missing indexes (check slow query log)
    2. Optimize N+1 query problems
    3. Increase connection pool size
    4. Consider read replicas for heavy read workloads
    5. Use database query caching
  </Accordion>
  
  <Accordion title="Connection pool exhaustion">
    **Symptoms:** "Timeout waiting for connection" errors
    
    **Solutions:**
    1. Increase pool size: `maximum-pool-size=50`
    2. Reduce connection timeout
    3. Fix connection leaks (check application code)
    4. Scale application horizontally
  </Accordion>
  
  <Accordion title="CORS errors in production">
    **Symptoms:** Browser blocks API requests
    
    **Solutions:**
    1. Verify `APP_CORS_ALLOWED_ORIGINS` includes your domain
    2. Check for typos (https vs http, www vs non-www)
    3. Ensure reverse proxy passes correct headers
    4. Test with curl: `curl -H "Origin: https://app.example.com" https://api.example.com/api/ping -v`
  </Accordion>
</AccordionGroup>
